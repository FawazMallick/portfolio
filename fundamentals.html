Finite Element Analysis

Why FEA Matters for Device Design

Every hardware component in the exoskeleton had to be structurally validated before anyone wore it. FEA lets you find failure modes in simulation — identifying where stress concentrations exceed material yield strength — before committing to fabrication. That's not just efficiency; for a wearable medical device, it's a safety requirement.

I learned FEA in depth through two frameworks: a 2D structural solver built on the BristolFE MATLAB library (which I used to understand the algorithm at the mesh, stiffness matrix, and solver level), and a 1D heat transfer solver written from scratch using the direct stiffness method.

2D Structural FEA Pipeline

The BristolFE framework implements the complete finite element pipeline — meshing, element stiffness assembly, global matrix construction, boundary condition application, linear system solve, and stress recovery. I used it to analyze structural components under load, learning the pipeline by working through it step by step.

1

Mesh Generation

Define geometry and element size. Auto-generate node positions and element connectivity. Hexagonal or rectangular structured meshes for simple geometries; unstructured Mesh2D for complex shapes.

2

Material & Stiffness Matrix

Define isotropic material (Young's modulus, Poisson's ratio). Compute element stiffness matrices using plane stress or plane strain formulations. Assemble into global K matrix.

3

Boundary Conditions & Loading

Apply displacement constraints at support nodes (frictionless base = vertical DOF fixed). Apply distributed stress or point displacement on loading surface.

4

Static Solve → Stress Recovery

Solve K·u = f for unknown displacements. Recover element stresses (σ_xx, σ_yy, σ_xy) from displacement field. Visualize with color-mapped contour plots.

1D Heat Transfer FEA from Scratch

For a thermodynamics assignment, I built a 1D heat transfer FEA solver from first principles using the direct stiffness method. The model: a three-layer composite (glass–air–glass) with conduction and convection terms, solved for nodal temperatures under applied heat flux.

Building this from scratch — assembling element stiffness matrices manually, handling conduction and convection terms, applying Neumann boundary conditions — gave me a much deeper understanding of what the structural FEA solver was doing under the hood.

What FEA taught me: The same mathematical structure — assemble element contributions into a global system, apply boundary conditions, solve — appears everywhere in numerical engineering. Understanding it from first principles means I can read the output of commercial FEA tools critically, not just accept them.

Biomechanics & Gait Analysis

Why Biomechanics Shows Up in My Robotics Work

You can't design a wearable device for a human without understanding how that human moves. The exoskeleton project required knowing what happens at the hip joint during a slip perturbation. The prosthetics research required understanding how gait phase maps onto sensor signals. These aren't optional extras — they're core design constraints.

Key Concepts Applied

  • Gait cycle phases: Stance (heel strike → toe-off) and swing (toe-off → heel strike). Phase detection from FSR and IMU is the foundation of both my exoskeleton and prosthetics work.
  • Ground reaction forces (GRF): The force the ground exerts on the foot. Changes in GRF pattern are early indicators of perturbations — used in exoskeleton control logic.
  • Joint torques: Calculated from inverse dynamics given GRF and kinematics. Validated against OpenSim simulation to set exoskeleton torque targets.
  • Foot placement mechanics: During swing phase, the nervous system uses sensory feedback to predict where the foot will land. My prosthetics research targets this prediction computationally.
  • Hip abductor function: The hip's lateral stabilization role during single-leg stance — the primary target of the exoskeleton actuator. Simulated in OpenSim under perturbation conditions.

OpenSim Musculoskeletal Simulation

For the exoskeleton project, I used OpenSim — the Stanford musculoskeletal simulation environment — to validate our control strategy before testing on humans. I built a full-body gait model, applied simulated slip perturbations during swing phase, and measured the stabilizing torque profile that our actuator needed to replicate.

The simulation revealed a critical constraint: the effective assistive torque window is only 100–200ms during early swing. Outside that window, intervention is either too late (the fall has begun) or counterproductive (disrupts normal gait). This directly set the real-time latency requirement for our IMU-based perturbation detection algorithm.

Embedded Systems & Sensors

The Hardware Side of Robotics

Algorithms only run on real hardware. For the exoskeleton project, I implemented the embedded sensing layer: reading orientation data from BNO055 IMUs over I²C, multiplexing multiple sensors on a shared bus using a TCA9548A multiplexer, and streaming real-time quaternion data to a PC for processing. This is the physical foundation the control algorithm runs on.

BNO055 IMU — 9-DOF Orientation Sensor

The BNO055 is a 9-axis absolute orientation sensor (accelerometer + gyroscope + magnetometer) with on-chip sensor fusion running at 100 Hz. It outputs calibrated quaternions directly — no need to implement complementary or Kalman filters for orientation estimation. This was critical for real-time gait phase detection in the exoskeleton.

Arduino — BNO055 Quaternion Reader (Actual IMU Code from Exoskeleton)
// BNO055 over I2C — quaternion output for real-time orientation tracking #include <Wire.h> #include <Adafruit_BNO055.h> Adafruit_BNO055 bno = Adafruit_BNO055(55); void setup() { Serial.begin(115200); Wire.begin(); if (!bno.begin()) { // check wiring if BNO055 not detected Serial.print("BNO055 not detected — check I2C wiring"); while(1); } delay(1000); // sensor calibration warmup } void loop() { imu::Quaternion q = bno.getQuat(); q.normalize(); // stream w, x, y, z to serial at 10 Hz for gait state detection Serial.print(q.w(), 4); Serial.print(" "); Serial.print(q.x(), 4); Serial.print(" "); Serial.print(q.y(), 4); Serial.print(" "); Serial.println(q.z(), 4); delay(100); }

Multi-Sensor I²C with TCA9548A Multiplexer

The exoskeleton required IMUs on both the thigh and shank — two BNO055 sensors that share the same I²C address (0x28). To put both on the same microcontroller bus, I integrated a TCA9548A I²C multiplexer, which routes communication to one of eight channels at a time. The tcaselect(channel) function writes to the multiplexer's control register before each sensor read, effectively switching which device is active on the bus.

This is a common pattern in multi-sensor embedded systems — one bus line, one address conflict, one multiplexer chip that solves it. Understanding this at the register level means I can adapt it to any I²C sensor combination.

TCA9548A Multiplexer — I²C Channel Select
// Select I2C channel on TCA9548A (address 0x70) void tcaselect(uint8_t channel) { Wire.beginTransmission(0x70); // multiplexer I2C address Wire.write(1 << channel); // bitmask selects one of 8 channels Wire.endTransmission(); } // Read both IMUs in loop — thigh on ch0, shank on ch1 for (uint8_t ch = 0; ch < 2; ch++) { tcaselect(ch); imu::Quaternion q = bno.getQuat(); q.normalize(); // process orientation per sensor... }

CAD & Fabrication

From Model to Physical Device

All design work on the exoskeleton started in SolidWorks — parametric CAD that let me iterate on bracket geometry, actuator mounting positions, and sensor housings without committing to material. Fabrication used the full workshop toolkit: laser cutting for flat aluminum brackets, 3D printing (ABS) for custom sensor housings, and a drill press for assembly.

We built three hardware iterations over the project timeline. Each iteration revealed something the simulation missed — a stress concentration, a fit issue, a sensor drift problem — and the CAD model was updated before the next fabrication run.

Design process insight: The hardest part of hardware isn't the CAD or the fabrication — it's the interface between the two. Where the 3D-printed part meets the laser-cut bracket is where tolerances accumulate. Where the IMU mount attaches to the brace is where sensor placement drift happens. Designing for fabrication variability is a skill that only comes from building things, not from simulation.

Why These Skills Matter Together

The reason I invested in these fundamentals is the same reason a good surgeon understands anatomy before picking up a robot: you can't trust a system you don't understand. FEA tells me whether my bracket will survive the loads I'm simulating in control theory. Biomechanics tells me which joint angles matter and why. The embedded systems work tells me whether my latency budget is physically achievable.

Research at the intersection of robotics, control, and medicine requires all three layers — the algorithm, the hardware, and the domain knowledge. These fundamentals are what connect them.
