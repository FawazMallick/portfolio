Featured Project: Turbofan Engine RUL Prediction

The Question: Can we predict when a jet engine will fail — before it does? Given multivariate sensor data from hundreds of engines across thousands of flight cycles, predict each engine's Remaining Useful Life (RUL) with enough accuracy to act on.

The Problem: Why RUL Matters

Turbofan engines degrade gradually. If you wait for failure, you get a catastrophic event. If you replace parts too early, you waste money. The goal is to predict — precisely — how many flight cycles remain before an engine needs maintenance. This is a safety-critical regression problem on noisy, multivariate time-series data.

I worked with the NASA C-MAPSS dataset — a benchmark for RUL prediction research. It simulates four fleets of 100–259 engines per fleet, with 21 sensor channels per time step, under varying operating conditions (one vs. multiple flight regimes and fault modes). Training was done on FD001 (single condition, 100 engines); generalization was tested on FD002–FD004 (multi-condition).

Dataset at a Glance

  • Training data (FD001): 20,631 time steps across 100 engines, 21 sensor channels
  • Test sets: FD001–FD004, ranging from 100 to 259 engines, single and multi-condition
  • Target: Cycles-to-failure for each engine at last observed time step
  • Challenge: FD002 and FD004 have six operating conditions — true generalization test
  • Infrastructure: Trained on Kaggle (2× NVIDIA GPU), ~100 epochs each model

Approach 1: Baseline GRU

A clean two-layer GRU that reads a sliding window of sensor observations and outputs a single RUL prediction. Trained end-to-end with a custom asymmetric scoring loss — the standard in RUL literature, which penalizes late predictions (missing imminent failure) more heavily than early ones.

GRU Architecture + Asymmetric Loss
class RUL_GRU(nn.Module): def __init__(self, input_size=21, hidden_size=64, num_layers=2): super().__init__() self.gru = nn.GRU(input_size, hidden_size, num_layers, batch_first=True) self.fc = nn.Linear(hidden_size, 1) def forward(self, x): out, _ = self.gru(x) return self.fc(out[:, -1, :]) # last time step → RUL scalar class AsymmetricScoringLoss(nn.Module): """Penalises late predictions (missed failures) harder than early ones.""" def forward(self, y_pred, y_true): d = y_pred - y_true return torch.where( d < 0, torch.exp(-d / 13) - 1, # under-prediction: steeper penalty torch.exp( d / 10) - 1 # over-prediction: milder penalty ).mean()

Approach 2: Embed-RUL (Encoder-Decoder Autoencoder)

A more complex, paper-inspired architecture. An encoder compresses each sensor window into a learned embedding space. A decoder reconstructs the window from that embedding — the autoencoder bottleneck forces the encoder to capture health-relevant structure. The embedding is then used to compute a Health Index (HI) curve per engine, which drives RUL estimation via a similarity-based matching algorithm across the training fleet.

The key insight: normal operating embeddings cluster together (near the learned Z_norm reference), while degraded embeddings drift away. Distance from Z_norm becomes a continuous degradation signal.

Encoder / Decoder Architecture
class Encoder(nn.Module): """Reads masked sensor windows → fixed-size health embedding.""" def __init__(self, input_size=45, hidden_size=55, dropout=0.2): super().__init__() self.cell = nn.GRUCell(input_size, hidden_size) self.drop = nn.Dropout(p=dropout) def forward(self, x): # x = [batch, seq, features+mask+delta] h = torch.zeros(x.size(0), self.hidden_size, device=x.device) for t in range(x.size(1)): h = self.cell(self.drop(x[:, t, :]), h) return h # final hidden = compact health embedding class Decoder(nn.Module): """Reconstructs sensor sequence from embedding (autoencoder objective).""" def forward(self, embedding, seq_len): h, outs = embedding, [] for _ in range(seq_len): h = self.cell(self.drop(h), h) outs.append(self.fc(h).unsqueeze(1)) return torch.cat(outs, dim=1)

Results

Both models were trained on FD001 and evaluated across all four test sets. Key findings:

Dataset Condition GRU RMSE GRU A(%) Embed-RUL RMSE Note
FD001 Single 13.17 60% 64.02 GRU wins; Embed shows degradation signal
FD002 Multi (6) 118.5 6.6% — Both struggle; distribution shift
FD003 Single 42.06 34% 70.25 GRU generalizes; Embed HI curves diverge
FD004 Multi (6) 124.5 6.6% — Multi-condition unseen at train time

Key Insight: The GRU baseline dominates on accuracy (RMSE 13.17 on FD001, paired t-test p < 0.0001). But its failure on multi-condition datasets reveals a fundamental limitation: a discriminative model trained end-to-end on one condition can't generalize without retraining. The Embed-RUL approach, while less accurate when it works, offers interpretability — the Health Index curves and t-SNE embeddings make it possible to see why the model predicts what it predicts, which matters in safety-critical deployment.

Takeaway: This project taught me that choosing a model architecture is an engineering decision, not just a performance optimization. Accuracy and interpretability trade off. Multi-condition generalization requires either domain adaptation or richer training data — not a bigger model.

Coursework Highlights

Across 13 modules, some assignments stood out for their depth. Each required implementing the algorithm from first principles — no sklearn wrappers until after proving you understood the math.

Module 3 · HW1

Logistic Regression + Polynomial Feature Mapping

Projectile hit/miss classification with a nonlinear decision boundary. Built a feature map that expands 2D inputs to 45 features through all 8th-order polynomial terms, then trained a logistic regression classifier with gradient descent.

The challenge: the raw input space has no linear separator. The feature mapping creates a rich enough function class that gradient descent finds a clean boundary. Implemented sigmoid, cross-entropy loss, its gradient, and the full training loop from scratch.

Result: >95% classification accuracy on held-out test set

Module 4 · HW1

Soft Margin SVM via Quadratic Programming

Implemented a soft-margin SVM classifier by directly solving the quadratic programming dual problem using cvxopt. Manually constructed the P, q, G, h constraint matrices from scratch.

The interesting part: sweeping over the regularization parameter C to watch how the margin and misclassification rate trade off. As C → 0, the classifier becomes increasingly tolerant of margin violations; as C → ∞, it approaches the hard-margin solution.

Solved QP directly — no sklearn SVC wrapper

Module 5 · HW1

Decision Tree vs Random Forest

Trained and compared a depth-10 decision tree against a 100-estimator random forest on the UCI Cylinder Bands dataset — 39 features, binary classification for rotogravure printer defects.

The decision tree achieved near-perfect training accuracy but lower test accuracy — classic overfitting. The random forest's ensemble averaging reduced variance and generalized substantially better. Wrote a clear analysis of why bootstrap aggregation works: decorrelated trees average out individual biases.

Random Forest: measurably higher test accuracy vs. single tree

Module 9–11

Sequence Models: RNNs, LSTMs, Attention

Implemented vanilla RNN backpropagation through time (BPTT), then LSTM with all four gate equations from scratch. Studied vanishing gradient problem experimentally — recorded gradient norms across time steps to see exactly where training signal collapses.

Attention mechanism implementation: scaled dot-product attention, then multi-head attention. This sequence directly prepared me for the GRU architectures in the RUL project and the encoder design in Embed-RUL.

Gradient norms visualized across sequence depths 10, 50, 100

Module 6–8

Backpropagation from Scratch

Implemented a fully-connected neural network with forward pass and manual backpropagation — computing weight gradients by hand via the chain rule, with no autograd. Then added batch normalization and dropout, verified numerically with gradient checking.

This exercise was harder than it sounds. Every matrix dimension mismatch is a bug in your math, not a bug in your code. Getting it right meant I actually understood what PyTorch's .backward() does under the hood.

Gradient check: numerical ≈ analytical to within 1e-7

Module 12–13

VAEs and Generative Models

Implemented a Variational Autoencoder (VAE) with the reparameterization trick. Minimized the ELBO loss combining reconstruction error with KL divergence. Generated new samples by sampling from the learned latent distribution.

This module connected directly to my research: the Embed-RUL encoder-decoder architecture is essentially a non-variational autoencoder applied to time-series. Understanding VAEs first made the Embed-RUL paper's design choices much clearer.

Applied to Real Problems

Beyond coursework, I've applied ML methods to two engineering domains that I care about — medical imaging and human gait. Both are harder than clean benchmark datasets: real-world noise, small data, and stakes that matter.

Fracture Detection with ResNet-50

For a biomedical engineering project, I applied a ResNet-50 convolutional neural network (transfer learning from ImageNet) to detect and localize fractures in X-ray images. The goal was to replicate surgical annotation — identifying fracture sites to support pre-operative planning.

We manually annotated a dataset of fracture X-rays, then fine-tuned ResNet-50 in MATLAB. The model was evaluated by comparing its predicted fracture regions against expert surgeon annotations. The side-by-side comparison showed high visual agreement — a promising proof of concept for AI-assisted surgical planning.

Bayesian Gait Prediction (Current RA Research)

My current research at CMU applies probabilistic ML — specifically Bayesian inference — to predict where a prosthetic leg user will place their foot next. Unlike the turbofan problem (clean, single-domain data), gait data is messy: sensor drift, inter-subject variability, latency constraints.

Why Bayesian? Because uncertainty quantification matters. In a safety-critical prosthetic control system, knowing "I'm 95% confident" is fundamentally different from "I'm 60% confident." A discriminative regression model gives a point estimate; a Bayesian model gives a distribution.

Current results: Offline evaluation on motion capture + IMU data shows RMSE < 50mm for foot placement prediction across multiple subjects and walking speeds. Feature importance analysis reveals that mid-swing angular velocity is the strongest predictor — consistent with biomechanical intuition about the swing-to-stance transition.

What This Builds Toward

The thread across all of this work is the same: building models that are right for their domain, not just models that score well on a benchmark. A GRU that achieves RMSE 13 on FD001 is impressive — until you realize it fails completely on FD002. A Bayesian model that gives uncertain predictions on a hard case is more useful than a neural network that confidently outputs the wrong answer.

I want to keep working at the intersection of ML and physical systems — where models have to interact with sensors, bodies, and environments that don't cooperate. The engineering constraints make the problems harder and the solutions more meaningful.
